# The Zen of JavaScript

_Inspired by Tim Peters; adapted for the Web._

- Expressive is better than restrictive.
- Explicit is better than "magic."
- Simple is better than clever.
- Clever is better than "I’ll just use a library for that."
- Modular is better than monolithic.
- Asynchronous is better than blocking.
- Performance counts, but DX (Developer Experience) keeps you sane.
- Edge cases will happen; handle them or they will haunt you.
- Although flexibility beats strictness.
- undefined is not null, and you should know why.
- Errors should be caught, not ignored.
- Unless you are intentionally letting it bubble up.
- In the face of this ambiguity, use an arrow function.
- There should be many ways to do it, but only one that your team agrees on.
- Although that way might change in the next TC39 proposal.
- Immutable is better than mutable.
- Although mutation is often faster in a tight loop.
- If the build tool is hard to configure, it’s a bad sign.
- If the code is easy to tree-shake, it’s a good sign.
- Closures are one honking great idea — let’s use them for everything!
